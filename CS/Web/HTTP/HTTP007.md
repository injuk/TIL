# HTTP
## 2026-05-23 Sat
### 쿠키란?
```
> 쿠키는 일종의 상태 정보이며, 서버에 의해 생성되어 클라이언트에게 전달된다.
> 클라이언트는 전달 받은 쿠키를 관리하며, 유효한 사이트에 접근할 때에만 전달한다.
```
- 쿠키는 사용자를 식별하고, 세션을 유지하기 위한 방식 중 가장 자주 사용되는 방식이다.
  - 이렇듯 쿠키는 중요할 뿐만 아니라, 쿠키를 위한 HTTP 헤더를 별도로 정의하기도 한다.
- 쿠키는 그 종류에 따라 크게 다음과 같이 구분할 수 있으며, **둘의 차이는 오직 파기 시점**에 있다.
  1. 세션 쿠키: 사용자가 사이트를 탐색하는 과정과 관련된 설정을 임시로 저장하며, 브라우저가 닫힐 경우 삭제된다.
  2. 지속 쿠키: 임시 쿠키와 달리 디스크에 저장되므로 브라우저가 닫히더라도 상태를 유지한다.

## 2026-05-24 Sun
### 쿠키의 동작 원리
```
> 쿠키의 기본적인 원리는 브라우저가 서버로부터 할당 받은 정보를 저장하고, 해당 사이트에 접근할 때마다 이를 함께 전송하는 것이다. 
```
- **쿠키는 서버가 클라이언트에게 할당하는 일종의 지시자**이며, 웹 사이트는 임의의 클라이언트에게 자신이 할당한 모든 지시자를 확인할 수 있다.
  - 기본적으로 쿠키에는 서버가 원하는 데이터를 무엇이든 할당할 수 있다.
- 클라이언트가 처음 웹 사이트에 접속할 경우, 서버는 클라이언트에 대해 아무런 정보도 갖지 못하므로 이를 식별할 수 있는 식별자를 쿠키에 할당한다.
  - 이 경우 응답의 `Set-Cookie` 또는 `Set-Cookie2` 헤더에 키-값 쌍 형태의 배열을 명시한다.
- **브라우저는 쿠키 데이터베이스를 관리하며, 사이트 별로 자신이 할당 받은 쿠키를 저장**한다.
  - 예를 들어 구글 크롬은 내부적으로 SQLite 기반의 데이터베이스를 관리하는 반면, 인터넷 익스플로러는 파일로 관리한다.
  - 때문에 추후 이미 방문했던 사이트에 재접속할 경우, 쿠키 데이터베이스에서 값을 찾아 `Cookie` 헤더에 명시한 후 요청을 전송한다.
  - 이렇듯 **브라우저는 쿠키를 저장할 책임을 가지며, 이를 `클라이언트 측 상태`라는 용어로 지칭**한다.
- 브라우저는 쿠키를 수백개 이상 저장할 수 있지만, 모든 사이트에 항상 쿠키 전부를 전송하지는 않는다.
  - 대신 해당 쿠키를 생성한 서버와 관련이 있는 쿠키들만을 요청 헤더에 명시하는 방식으로 동작한다.
  - 이는 성능 저하를 막기 위함이며, 각 사이트마다 자신이 활용할 수 있는 쿠키가 다르기 때문이다.

## 2026-05-25 Mon
### 쿠키 적용 범위 제어하기
- 서버는 쿠키 생성시 전달하는 `Set-Cookie` 헤더에 `domain` 속성을 작성하는 것으로 어떤 사이트가 해당 쿠키를 조회할 수 있는지 제어할 수 있다.
  - 예를 들어, `Set-Cookie: user="injuk"; domain="example.com"`과 같이 명시할 수 있다.
  - 이 경우, 호스트와 관계 없이 `example.com`으로 끝나는 사이트에 접속할 경우 항상 `Cookie: user="injuk"`가 전달된다.
- 반면, `path` 속성에 URL의 일부를 명시하는 것으로 해당 경로에 접속할 때에만 쿠키를 명시하도록 제어할 수도 있다.
  - 예를 들어, `Set-Cookie: user="injuk"; domain="example.com"; path=/study/`와 같이 명시할 수 있다.
  - 이 경우, `example.com/study/` 경로로 시작하는 모든 리소스에 대해 쿠키를 명시하게 된다.

## 2026-05-26 Tue
### 그 밖의 쿠키 구성 요소
- 쿠키는 크게 넷스케이프 쿠키라는 용어로 지칭되는 Version 0 쿠키와 Version 1 쿠키로 구분될 수 있다.
  - 반면, Version 1 쿠키는 Version 0에 대한 확장이지만 널리 보급되지는 못했다.
- Version 0 쿠키를 예로 들어, `domain`과 `path` 외에도 다음과 같은 속성을 명시할 수 있다.
  1. `expires`: 선택적으로 입력하여 쿠키의 생명주기를 명시할 수 있으며, 만료일에 다다르면 해당 쿠키는 삭제되어 전달되지 않는다.
  2. `secure`: 역시 선택적인 속성으로, `Set-Cookie: id=1; secure` 형태로 명시할 경우 SSL 기반의 보안 연결 시에만 해당 쿠키를 명시한다.
- Version 1 쿠키의 경우, 이에 더해 `comment`나 `discard` 등의 여러 속성이 추가되었다.

## 2026-05-27 Wed
### 쿠키와 버전 협상
- `Cookie2` 요청 헤더는 서로 다른 쿠키 버전을 지원하는 클라이언트와 서버 사이의 협상에 사용된다.
  - 예를 들어, 클라이언트는 `Cookie2: $Version="1"`과 같이 명시하는 것으로 자신이 새로운 버전의 쿠키를 지원함을 서버에게 알릴 수 있다.
- 서버는 새로운 형식의 쿠키를 인식할 경우 `Set-Cookie`가 아닌 `Set-Cookie2` 헤더를 활용해야 한다.
  - 반면, 클라이언트가 `Cookie`와 `Cookie2`를 모두 명시했다면 이전 버전인 `Cookie` 만을 무시해야 한다.
- 클라이언트가 두 버전을 모두 지원할 수 있다고 하더라도 서버가 `Set-Cookie`로 응답했다면 Version 0 쿠키를 사용해야 한다.

  2. 본문은 캐시하되 `Set-Cookie` 헤더를 캐시하는 것은 주의해야 한다.
  3. `Cookie` 헤더에는 개인정보가 포함될 수 있으므로 캐싱에 주의해야 한다.

### 쿠키와 보안
- 쿠키라는 기술 자체가 개인정보를 다루거나 사용자를 추적하는 데에 사용될 수 있으므로 부정적인 여론을 갖고 있는 경우가 많다.
  - 그러나 쿠키는 비활성화하거나, 로그 분석 등의 방법으로 대체하는 것이 가능하므로 보안상 엄청나게 위험한 기술로 볼 수는 없다.
  - 애초에 쿠키에 개인정보를 저장하는 것이 아닌, 이를 백엔드 상에서 관리하고 그 키 값만을 주고 받는다면 예민한 데이터를 주고받는 상황을 줄일 수 있다.
- 쿠키는 이에 명시할 개인정보를 누가 전달받을지 명확히 하고, 사이트의 개인정보 정책을 신경쓴다면 위험성보다 편리함이 주는 이점이 더 크다.

### 인증의 필요성
- 인증은 자신이 누구인지 증명하는 것으로, 현실의 주민등록증 또는 여권 기반 인증을 예로 들 수 있다.
- 많은 사람이 웹을 통해 개인 정보를 다루므로, 웹의 정보 일부는 허가된 사람만이 접근할 수 있어야 한다.
  - 예를 들어, 개인 프로필이나 개인적인 문서 등은 소유자의 동의 없이는 확인할 수 없어야 한다.
- 이를 위해 서버는 사용자를 정확히 식별할 수 있어야 하며, 이를 통해 사용자가 어떤 작업이나 리소스에 대해 접근 가능한지 판단이 가능하다.
  - 이러한 일련의 과정은 `인증`이라는 용어로 표현할 수 있으며, 일반적인 인증 메커니즘은 이름과 비밀번호 방식이다.
- HTTP는 자체적인 인증 기능을 제공하며, 크게 기본 인증과 다이제스트 인증으로 구분할 수 있다.
  - 반면, 최근의 웹 사이트들은 보안을 강화하기 위해 HTTP 인증보다는 인증 모듈을 직접 구현하는 방식을 택한다.

## 2026-05-31 Sun
### HTTP 인증 프레임워크의 동작 원리
```
> HTTP는 기본 인증과 다이제스트 인증이라는 이름의 두가지 공식 인증 프레임워크를 제공한다.
```
- HTTP의 경우 사용자 인증을 위해 자체적인 인증 요구 / 응답 프레임워크를 제공하며 이는 다음과 같이 동작한다.
  1. 클라이언트가 서버에 저장된 민감한 정보에 대한 접근을 요청한다.
  2. 서버는 `401` 상태 코드로 응답하는 것으로 인증을 요구한다.
  3. 클라이언트는 1.의 정보에 다시 접근하되, 사용자의 인증을 마친 후 인증 정보를 서버에 전달한다.
  4. 서버는 사용자에 대한 인가 작업을 거친 후, 적절한 권한을 갖는 사용자일 경우에만 민감한 정보를 반환한다.
- 반면, 클라이언트가 제출한 인증 정보가 적절하지 않은 경우 서버는 재인증을 요구하거나 에러를 반환할 수 있다.

## 2026-06-01 Mon
### HTTP 기본 인증 동작 원리
```
> 기본 인증은 널리 알려진 HTTP 규약이며, 거의 모든 클라이언트와 서버는 기본 인증을 구현한다.
```
- 기본 인증 과정에서 서버는 클라이언트로부터의 접근에 대해 `401` 상태 코드와 `WWW-Authenticate` 응답 헤더를 반환할 수 있다.
  - 이 때, 응답 헤더에는 기본 인증을 의미하는 `Basic`과 리소스의 보안 영역을 의미하는 `realm`이 명시될 수 있다.
  - 예를 들어 하나의 웹 서버가 리소스를 여러 논리적인 영역으로 구분하는 경우, `realm` 지시자를 활용하여 원하는 리소스에 대한 인증 방식을 결정할 수 있다.
- 이에 대해 브라우저는 사용자에게 이름과 비밀번호 입력을 요구하며, 입력 받은 정보를 암호화한 후`Authorization` 헤더에 명시하여 서버에 전송한다.
  - 이 때, 인증 정보는 `이름:비밀번호` 형태로 결합된 후 `BASE64` 방식으로 인코딩한 후에 전송한다.
- 또한, 기본 인증의 경우 `Authentication-Info`와 같이 인증과 관련된 추가 정보를 요구하지 않는다.

### 중개 프록시 기반의 인증
- 일부 기업은 중개 프록시 서버를 두고, 이를 인증에 활용하는 방식을 택하기도 한다.
  - 예를 들어, 사내 서버나 LAN 또는 무선 네트워크에 접근하는 경우 프록시 서버를 거치게 하여 인증을 요구하는 방식으로 구성할 수 있다.
- 이러한 인증 전용 중개 프록시 서버는 접근 정책을 중앙 관리할 수 있기에 사내에 통합적인 접근 제어 방식을 적용할 수 있다.
- 이 경우, 인증의 첫 단계는 `407` 상태 코드를 기반으로 한 프록시 인증을 요구하는 데에 있다.
  - 또한, 프록시 기반의 인증 방식은 `Proxy-Authenticate`와 같이 `Authenticate` 접두사가 명시된 헤더를 사용한다.

## 2026-06-03 Wed
### 기본 인증의 한계
- 기본 인증은 단순하지만 다음과 같은 이유에서 안전하다고 볼 수 없다.
  1. `BASE64`는 누구나 사용할 수 있으므로, 악의적인 사용자가 정보를 디코딩하기 쉽다.
  2. 기본 인증 정보가 탈취된 경우 악의적인 사용자에 의해 악용될 수 있다.
  3. 어떤 사용자는 모든 사이트에 동일한 이름 및 비밀번호를 사용할 수도 있다.
  4. 트랜잭션의 의도를 바꾸는 프록시 등이 배치되어 있는 경우, 기본 인증은 정상 동작을 보장할 수 없다.
  5. 기본 인증은 악의적인 서버가 개입되는 시나리오에 취약하다.
- 이렇듯 기본 인증은 보안에 민감하지 않은 환경에서는 편리하지만, 보안에 민감한 상황에는 적절하지 못하기에 SSL 등과 같은 기술과 연계되어야 한다.

## 2026-06-04 Thu
### 다이제스트 인증이란?
- 앞서 다룬 기본 인증은 편리하지만 전혀 안전하지 않으며, 메시지를 보호하려는 어떠한 장치도 존재하지 않는다.
- 이에 다이제스트 인증은 기본 인증과 호환되면서도 안전한 대체제로서 기능할 수 있도록 제안되었으며, 다음과 같은 특징을 갖는다.
  1. 네트워크 상 비밀번호를 평문으로 전송하지 않는다.
  2. 인증 체결을 가로채려는 악의적인 사용자들을 차단한다.
  3. 메시지 내용의 위변조를 방지한다.
- 다이제스트 인증은 가장 안전한 프로토콜은 아니지만, 적어도 기본 인증보다는 훨씬 강력하다고 볼 수 있다.
  - 반면, 보안에 더 적합한 프로토콜로는 TLS와 HTTPS를 예로 들 수 있다.

## 2026-06-05 Fri
### 다이제스트 인증의 동작 원리
```
> 다이제스트 인증의 핵심은 '절대 비밀번호를 네트워크에 전송하지 않는다'는 점에 있다.
```
- 다이제스트 인증은 비밀번호를 네트워크 상에 전송하는 대신 이를 비가역적으로 뒤섞은 지문, 또는 요약을 전송한다.
- 이는 클라이언트와 서버가 모두 비밀번호를 알고 있으므로, 전송된 지문이 이와 일치하는지 판별할 수 있다는 점에서 출발한다.
  - 예를 들어 클라이언트는 비밀번호에 대한 지문을 생성하고 전송하며, 서버는 자신이 갖는 비밀번호에 대한 지문을 생성하여 비교한다.

- 이 때, 지문 또는 요약 정보는 본문에 대한 단방향 압축이며 입력 가능한 모든 값을 유한한 범위의 압축으로 변환하는 방식으로 동작한다.
  - 예를 들어, 유명 알고리즘인 `MD5`는 임의의 바이트 배열을 128 비트로 변환한다.

  2. 클라이언트 인증: 위조된 사용자가 아님을 알 수 있어야 한다.

  4. 암호화: 도청에 대한 걱정 없이 트랜잭션을 처리할 수 있어야 한다.
  5. 효율: 충분히 빠른 알고리즘을 제공해야 한다.
  6. 편재성: 거의 모든 클라이언트와 서버에서 지원 가능해야 한다.

## 2026-06-08 Mon
### HTTPS란?
```
> HTTPS는 HTTP에 대한 보안 기능 중 가장 유명한 기능이다.
```
- HTTPS는 `https://` 스킴을 사용하며, 모든 HTTP 요청과 응답이 네트워크에 전달되기 전에 암호화한다.
- 이러한 HTTPS는 HTTP 하부에 전송 레벨의 암호 보안 계층을 제공하는 것으로 동작한다.
  - 다시 말해, TCP/IP 계층도 상 전송 계층과 애플리케이션 계층 사이에 보안 계층을 추가한다.
  - 이 때, 인코딩 및 디코딩 작업 전반은 보안 계층에서 처리하므로 보안 기능을 누리기 위해 HTTP 애플리케이션을 크게 수정할 필요는 없다.
- HTTPS는 암호화 기술을 기반으로 하기에, 디지털 암호학에 대한 배경 지식이 필요할 수 있다.

### HTTPS와 HTTP의 차이
```
> HTTPS는 HTTP의 가장 유명한 보안 버전이며, 널리 구현되어 주요 상용 브라우저 및 서버에 적용되어 있다.
```
- HTTPS는 HTTP 프로토콜에 대칭키 및 공개키 암호화, 그리고 인증서와 같은 강력한 암호 기법을 적용한 개념이다.
  - 이러한 기능 집합을 통해 혼란한 웹 환경에서도 HTTP를 안전하게 사용할 수 있다.
- HTTPS는 보안 전송 계층을 활용하는 HTTP이며, 암호화되지 않은 HTTP 메시지를 TCP 세그먼트로 전환하기 전에 암호화 계층을 적용한다.
- 그러나 보안 HTTP의 사용은 선택적이므로 웹 서버에 요청할 때 보안 기능을 활성화할지 알려줄 수 있는 방법이 필요하며, 이는 URL의 스킴 정보를 통해 구분된다.
  - 예를 들어, `https` 스킴을 사용하여 서버에 질의한 경우 보안 HTTP를 사용하고자 하는 뉘앙스를 포함한다.
- 보안 HTTP를 수립하는 SSL 핸드셰이크 과정에서 서버의 인증서가 전달되며, 클라이언트는 이를 검증해야 한다.
  - 이 경우, 클라이언트는 인증서의 날짜와 서명 기관의 신뢰도 및 디지털 서명 검사를 차례로 수행한다.

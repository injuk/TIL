  2. 새로 생성된 페이지 목록
  3. 자주 링크되지 않는, 잘 알려지지 않은 페이지 목록
- 대부분의 대규모 크롤러는 루트 집합에 새 페이지나 잘 알려지지 않은 페이지를 추가하는 기능을 제공하며, 이들 루트 집합은 시간에 따라 성장하는 경향을 보인다.

## 2026-05-02 Sat
### 크롤링과 순환 회피
- 크롤러는 동작 과정에서 꾸준히 HTML 문서를 수집하고, 해당 문서 내에 포함된 URL 링크를 파싱하여 크롤링 대상 페이지 목록에 추가한다.
  - 이 때, 크롤러는 HTML 파싱을 통해 링크를 추출하고 상대 링크를 절대 링크로 변환한다.
- 크롤러의 동작 과정에서 단연 중요한 것은 루프나 순환에 빠지지 않는 것으로, 이는 크게 다음과 같은 이유로 인해 해롭다고 볼 수 있다.
  1. 순환: 크롤러가 루프에 빠져 정상적으로 동작하지 않을 수 있다.
  2. 부하: 크롤러가 임의의 페이지를 계속해서 요청할 경우, 해당 페이지를 제공하는 서버의 부담이 된다.
  3. 중복: 루프 자체가 큰 문제가 되지 않더라도, 크롤러는 수많은 무의미한 중복 컨텐츠를 반환할 수 있다.
- 때문에 크롤러는 순환을 회피해야 하며, 수십 억개 이상의 URL에 대한 방문 기록을 유지하기 위해 다음과 같은 기법을 고려할 수 있다.
  1. 트리 및 해시 테이블: 방문한 URL을 빠르게 조회할 수 있도록 하는 자료 구조에 해당한다.
  2. 느슨한 존재 비트맵: 방문한 URL은 해시 함수에 의해 고정된 크기의 숫자로 변환되고, 배열 내에 대응하는 존재 비트를 두어 검증한다.
  3. 체크포인트: 크롤러가 예상치 못하게 중단된 경우에 대비하여 방문한 URL 목록을 디스크에 저장한다.
  4. 파티셔닝: 웹이 성장함에 따라 하나의 크롤러가 크롤링을 완수하는 것은 불가능해졌기에, 여러 크롤러로 구성된 `farm`을 통해 크롤링을 수행한다.

## 2026-05-03 Sun
### 순환을 어렵게 만드는 요소들
- 상술한 자료 구조를 갖추었더라도 다음과 같은 이유에서 크롤러는 손쉽게 순환에 빠질 수 있다.
  1. URL 별칭: 포트 번호의 누락 여부나 URL 정규화 여부에 따라 동일한 URL이 여러 형태를 가질 수 있다.
  2. 파일 시스템 링크 순환: 웹 서버의 파일 시스템에 심볼릭 링크 등을 활용한 재귀 참조가 있을 수 있다.
  3. 동적 가상 웹 공간: 가상 URL에 대해 요청받은 웹 서버가 동적으로 HTML을 생성하는 경우가 있을 수 있다.
- 반면, 모든 순환을 피할 수 있는 이상적인 방법은 존재하지 않으며 오히려 좋은 크롤러는 순환 회피를 위한 휴리스틱 집합을 갖는다.
- 이렇듯 웹에는 크롤러가 문제를 일으킬 수 있을만한 이유가 너무 많기에, 크롤러가 정상 동작할 수 있도록 사용하는 다음과 같은 기법들이 존재한다.
  1. URL 정규화
  2. 너비 우선 크롤링
  3. 스로틀링
  4. URL 크기 제한
  5. 사이트 블랙 리스트
  6. 패턴 감지
  7. 컨텐츠 지문
- 반면, 상술한 기법들로도 해결할 수 없는 문제에 봉착하는 경우에 대비하기 위해서는 결국 사람이 문제를 빠르게 인지하고 개입할 수 있도록 해야 한다.

## 2026-05-04 Mon
### 웹 로봇과 HTTP - 필수 헤더
- 웹 로봇 역시 여타 흔한 HTTP 앱과 다르지 않으며, HTTP 명세를 준수할 책임이 있다.
  - 그러나 일반적으로 `HTTP/1.1`보다 `HTTP/1.0`에서 요구되는 규칙이 적기에 웹 로봇은 `HTTP/1.0`을 사용하는 경향이 있다.
- 아무리 웹 로봇이 최소 구현을 선호한다고 해도, 대부분의 경우 HTTP 헤더 중 신원 식별과 관련된 다음의 헤더는 필수로 구현한다.
  1. `User-Agent`: 웹 서버에게 웹 로봇의 이름을 알린다.
  2. `From`: 웹 로봇의 사용자, 또는 관리자의 이메일 주소를 제공한다.
  3. `Accept`: 웹 로봇이 어떤 미디어 타입을 처리할 수 있는지 명시한다.
  4. `Referer`: 현재 요청 URL을 포함하는 문서의 URL을 제공한다.
- 반면, 웹 로봇의 구현자는 가상 호스팅을 사용하는 서버에 대비하여 `Host` 헤더를 지원할 필요가 있다.
  - 그렇지 않은 경우, 가상 호스팅을 사용하는 서버에 대해 잘못된 정보를 수집할 가능성이 높다.

### 웹 로봇과 HTTP - 응답 처리
- 일부 웹 로봇은 매우 많은 양의 요청을 시도하므로, 검색 대상 컨텐츠의 양을 줄이기 위해 조건부 HTTP 요청을 구현하여 캐시를 적극적으로 활용한다.
- 대부분의 웹 로봇은 `GET` 메소드를 사용하는 반면, 조건부 `GET` 등을 사용하는 경우가 있으므로 응답 상태 코드를 적절히 처리할 수 있어야 한다.
  - 기본적으로 `200`과 `404`를 필수적으로 이해해야 하며, 예외 상황에 `200`으로 응답하는 서버 역시 처리할 수 있도록 구현하는 것이 바람직하다.
- 일부 서버는 `HTML`내의 `HEAD` 태그에 응답 헤더를 덮어 쓰는 내용을 작성할 수 있으므로, 웹 로봇 구현시 필요에 따라 해당 태그를 처리할 수 있어야 한다.
- 반면, 웹 서버 관리자의 경우 많은 웹 로봇이 자신의 사이트에 방문하게 될 것임을 예상할 수 있어야 한다.
  - 다시 말해, 관리자는 웹 로봇의 요청을 처리하기 위한 전략을 수립해야 ㅎ한다.
  - 예를 들어, 풍부한 기능을 갖지 않는 웹 로봇이 요청했을 때에도 잘 대응할 수 있는 페이지를 개발할 수 있다.

### 바람직하지 않은 웹 로봇
- 자동으로 동작하는 웹 로봇 특성상 실수를 저지르기 쉬우며, 그 대표적인 예시를 다음과 같이 들어볼 수 있다.
  1. 순환에 의한 폭주: 웹 로봇은 HTTP 요청을 사람보다 빠르게 생성하므로, 순환에 빠진 경우 네트워크에 큰 부하를 줄 수 있다.
  2. 오래된 URL: 오래된 URL 목록을 사용하는 웹 로봇의 경우, 존재하지 않는 URL에 대한 요청을 자주 발생시킬 수 있다.
  3. 잘못된 URL: 오류로 인해 웹 로봇이 임의의 웹 서버에게 길고 무의미한 URL에 대한 요청을 자주 발생시킬 수 있다.
  4. 사생활 침해: 웹 로봇이 사적인 데이터에 대해 접근하는 경우, 사생활 침해 문제로 불거질 수 있다.
  5. 동적 게이트웨이 접근: 처리 비용이 큰 게이트웨이 애플리케이션의 컨텐츠를 의미하는 URL에 접근하는 경우, 웹 서버에 부하를 줄 수 있다.
- 사생활 침해의 경우, 사적인 컨텐츠에 대해 존재하는 링크를 웹 로봇이 따라가는 것으로 발생하기 쉬우므로 민감한 데이터를 무시하도록 구현하는 것이 바람직하다.
  - 반면, 엄밀히 말해 링크가 존재하여 공개된 인터넷을 통해 접근 가능한 리소스는 사적인 것으로 볼 수 없기도 하다.
- 웹 서버의 관리자는 상술한 웹 로봇의 오동작에 대비하여 특정 컨텐츠에는 웹 로봇이 접근할 수 없도록 제어할 것을 고려해볼 수 있다.

### 웹 로봇 차단하기
- 웹 로봇 개발자들은 로봇이 유발할 수 있는 문제들을 잘 알고 있으며, 이에 관리자들이 로봇의 동작을 제어할 수 있는 메커니즘을 자발적으로 개발하여 제공했다.
  - 이는 번듯한 공식 명칭이 있는 표준이나, 일반적으로는 관련된 파일의 이름을 따 `robots.txt`라고 지칭한다.
- 웹 로봇의 동작을 제어하기 위한 서버는 문서 루트에 `robots.txt` 파일을 둘 수 있으며, 해당 파일에 어떤 로봇이 어떤 리소스에 접근할 수 있는지 명시한다.
  - 자발적으로 표준에 따르는 웹 로봇은 임의의 리소스에 접근하기 전에 우선 해당 파일을 요청하는 방식으로 동작한다.
  - 만약, 해당 리소스에 접근하는 것이 허용된다는 사실이 `robots.txt`에 명시되어 있다면 웹 로봇은 리소스에 실제로 접근하게 된다.
- 그러나 차단 표준은 임시방편으로 제작된 성향이 강하고, 이를 소유하는 주체가 없다는 한계가 있다.
  - 때문에 웹 로봇을 개발하는 여러 업체는 해당 표준의 전체가 아닌 부분 집합을 구현하는 성향이 강했다.
  - 이렇듯 웹 로봇의 동작을 제어하는 기능은 불완전하지만, 그조차 없는 것보다는 훨씬 낫기에 대부분의 업체와 크롤러는 이들 표준을 지원한다.
- 웹 로봇 차단 표준은 `v0.0`과 `v1.0`, `v2.0`이 있으나 `v2.0`이 널리 채택되지 못한 탓에 대부분의 웹 로봇은 `v0.0`과 `v1.0` 표준을 준수한다.

## 2026-05-08 Fri
### 웹 로봇과 robots.txt 파일 조회
- 웹 로봇이 임의의 사이트에서 제공하는 어떤 리소스에 접근하기 전에 로봇은 해당 사이트의 `robots.txt`를 조회하여 처리할 책임이 있다.
  - 이 때, 임의의 호스트 명과 포트 번호에 의해 정의된 웹 사이트는 최대 하나의 `robots.txt` 파일만을 갖는다.
  - 반면, 가상호스팅된 경우 각각의 가상 `docroot`에 서로 다른 `robots.txt` 파일이 포함될 수 있다.
- 웹 로봇은 `GET` 메소드로 해당 파일을 요청하며, 서버는 파일이 존재하는 경우 `text/plain` 형태의 본문을 반환한다.
- 이 때, 웹 로봇은 웹 서버의 응답에 따라 다음과 같이 가정한 후에 동작해야 한다.
  1. `200`: 웹 로봇은 해당 파일의 내용을 파싱하여 규칙을 획득한 후, 해당 규칙에 따라 접근해야 한다.
  2. `3xx`: 웹 로봇은 실제 리소스가 발견될 때까지 리다이렉트를 따라가야 한다.
  3. `404`: 해당 서버에 활성화된 규칙이 없다고 가정해도 무방하다.
  4. `401` 또는 `403`: 해당 서버로의 접근이 완전히 제한된다고 가정해도 무방하다.
  5. `503`: 요청 시도에 일시적으로 실패한 것으로 취급하고, 해당 사이트로의 접근을 뒤로 미룬다.
- 반면, `robots.txt` 파일은 사이트 관리자에 의해 관리되므로 컨텐츠 작성자 개개인은 웹 로봇의 접근을 제어할 수 없다는 한계가 있다.
  - 때문에 HTML 저자가 웹 로봇의 접근을 제어할 수 있도록 하는 `META` 태그를 사용할 수 있으며, 웹 로봇은 표준과 마찬가지로 이에 따를 것이 권장된다.
  - 예를 들어, `<META NAME="ROBOTS" CONTENT="NOINDEX">`와 같이 명시할 수 있으며 이는 HTML의 `HEAD` 섹션에 명시되어야 한다.

### robots.txt 파일 양식
- 해당 파일은 줄 기반의 단순한 문법으로 작성되며, 각 줄은 빈 줄이거나 주석이거나 규칙일 수 있다.
  - 이 때, 규칙 줄은 HTTP 헤더 형태의 키-값 쌍으로 작성된다.
- 여러 개의 규칙 줄을 모은 규칙 집합을 레코드라는 용어로 지칭하며, 각 레코드는 비 줄이나 파일 끝 문자로 끝난다.
  - 각각의 레코드는 어떤 웹 로봇이 해당 레코드에 영향을 받는지 명시하는 하나 이상의 `User-Agent` 규칙을 명시한다.
  - 그 뒤로는 임의의 리소스에 대한 허용 여부를 설정하는 `Allow` 또는 `Disallow` 규칙을 명시한다.
- `User-Agent` 규칙은 웹 로봇의 이름 일부 또는 `*`을 명시할 수 있으며, 웹 로봇이 전송한 `GET` 요청에 명시된 `User-Agent` 헤더를 검증한다.
  - 웹 로봇은 대소문자와 관계 없이 자신 이름의 부분 문자열이 될 수 있는 레코드 중 첫 번째 또는 `*`이 명시된 레코드 중 첫 번째 항목을 준수해야 한다.
  - 반면, 웹 로봇이 적절한 레코드를 찾지 못한 경우 해당 웹 서버로의 접근에는 어떠한 제한도 적용되지 않는다.
- `Allow` 또는 `Disallow` 규칙은 `User-Agent` 규칙 다음에 명시되며 임의의 웹 로봇이 어떤 리소스에 대해 명시적으로 허용 / 거부되는지를 의미한다.
  - 대응되는 여러 규칙이 있는 경우, 부합하는 첫 번째 규칙에 따르는 반면 그 무엇에도 부합하지 않는 경우에는 허용으로 간주한다.
  - 또한, 해당 규칙에 명시된 리소스는 접두 매칭을 적용하므로 빈 문자열을 명시할 경우 모든 경로에 매칭된다.
  - 반면, `robots.txt` 파일에 대한 접근은 항상 허용하기 위해 해당 규칙에 명시되지 않아야 한다.

### robots.txt 주의사항
- 해당 파일의 명세 역시 시간이 지남에 따라 발전할 수 있으므로, 상술한 규칙 외에 다른 필드가 포함될 수 있음에 주의해야 한다.
  - 예를 들어 웹 로봇은 자신이 이해하지 못하는 필드를 무시하는 식으로 동작한다면 별다른 문제가 발생하지 않는다.
  - 대표적으로, 웹 로봇 차단 표준의 `v0.0`은 `Allow` 규칙을 지원하지 않기에 해당 규칙을 무시한다.
- 하위 호환성을 보장하기 위해 하나의 줄을 여러 줄로 나누어 작성하는 것은 허용되지 않는다.
- 주석은 파일의 어느 위치에서든 허용되어야 하며, 주석 문자로 시작하여 처음으로 마주하는 개행 문자까지 주석으로 취급해야 한다.

### robots.txt 파일 캐싱하기
- 매 접근시마다 해당 파일을 조회해야 한다면 웹 로봇의 성능과 효율성이 떨어지게 되며, 웹 서버에 가해지는 부담도 늘어난다.
  - 이에 대비하여 웹 로봇은 해당 파일을 캐싱할 수 있으며, 이 과정에서 HTTP의 캐시 제어 메커니즘이 적용될 수 있다.
- 반면, 많은 웹 로봇은 `HTTP/1.1` 클라이언트가 아니므로 여러 캐시 제어 메커니즘을 이해할 수 없는 경우가 있다.
  - 때문에 웹 서버 관리자는 이러한 상황을 기억하고 충분히 주의를 기울일 필요가 있다.

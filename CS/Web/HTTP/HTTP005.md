# HTTP
## 2026-03-27 Fri
### 웹 캐시란?
- 웹 캐시는 자주 사용되는 문서의 사본을 자동으로 보관하는 HTTP 장치를 의미한다.
  - 새로운 웹 요청이 캐시에 도달했을 때, 캐시에 적절한 사본이 존재한다면 원 서버에 요청하지 않을 수 있다.
- 캐시는 IO 작업의 횟수에 직접적인 영향을 줄 수 있으며, 적절히 설정된 경우 다음과 같은 이점을 제공한다.
  1. 불필요한 데이터 전송 방지
  2. 네트워크 병목 감소
  3. 원 서버 부담 감소
  4. 리소스 요청시 응답 지연 감소

## 2026-03-28 Sat
### 캐시 히트와 캐시 미스 및 재검사
- 캐시는 유용하지만, 세상 모든 문서의 사본을 저장할 수는 없으므로 부득이하게 필요한 문서만을 캐시하게 된다.
- 이 때, 임의의 요청에 대응되는 사본이 존재하는 경우를 캐시 히트라고 지칭할 수 있다.
  - 반면, 사본이 존재하지 않아 원 서버로 요청하는 경우는 캐시 미스가 된다.
- 그러나 원 서버의 리소스가 변경될 수 있으므로 캐시는 사본이 최신인지 서버를 통해 점검할 필요가 있으며, 이를 재검사라는 용어로 지칭한다.
  - 재검사 요청의 효율성을 높이기 위해, HTTP는 서버로부터 전체 객체를 가져오지 않고 재검사할 수 있도록 하는 특별한 요청을 정의한다.
  - 캐시는 임의의 사본에 대한 재검사가 필요한 경우 원 서버에 작은 요청을 전송하며, 서버는 변경되지 않은 경우에 `304`로 응답할 수 있다.
- HTTP는 캐시 재검사를 위해 `If-Modified-Since` 헤더를 제공하며, 서버는 해당 시간 이후에 리소스가 변경된 경우에만 사본을 전송한다.
  - 서버의 입장에서, 변경되지 않은 경우 `304`로 응답한다.
  - 변경된 경우, 서버는 `200`을 통해 리소스 전체를 반환한다.
  - 반면, **서버에서 해당 리소스가 삭제된 경우 `404`를 반환하게 되므로 캐시는 사본을 삭제**할 수 있다.
- 반면, HTTP는 클라이언트에게 캐시가 적중했는지 알려줄 수 있는 별도의 방법을 제공하지는 않는다.
  - 클라이언트는 대신 `Date` 헤더나 `Age` 헤더를 통해 응답의 캐싱 여부를 추측할 수 있다.

### 히트율과 바이트 적중률
- 캐시가 요청을 처리하는 비율을 히트율(Hit ration)이라고 표현하며, 일반적으로 40%의 히트율을 보장할 경우 좋은 캐시로 취급할 수 있다.
  - 이를 개선할 경우, 외부로 나가는 HTTP 트랜잭션이 줄어들어 전체 지연이 감소한다.
- 반면, 바이트 적중율은 캐시를 통해 트래픽이 절감된 양을 측정하는 데에 초점을 맞춘다.
  - 이를 개선할 경우, 외부로 나가는 바이트의 양이 줄어들어 대역폭을 크게 절약할 수 있다.

## 2026-03-30 Mon
### 캐시 토폴로지
- 캐시는 공유 대상의 규모에 따라 크게 다음과 같이 구분할 수 있다.
  1. 개인용 캐시: 브라우저 캐시와 같이 사용자 한 명을 위한 작고 저렴한 캐시를 의미한다.
  2. 공용 프록시 캐시: 캐싱 기능을 제공하는 일종의 프록시이며, 여러 사용자가 접근할 수 있다.
- 이 때, 작고 저렴한 캐시에서 캐시 미스가 발생했을 때 프록시 연쇄에 따라 더 성능이 좋은 부모 캐시가 이를 처리하도록 구현할 수 있다.
  - 이는 일종의 캐시 계층으로, 둘 혹은 세 개의 캐시 프록시 연쇄를 두는 합리적인 방식에 해당한다.
- 반면, 캐시 계층보다 더 복잡한 형태의 캐시망을 구현할 수도 있으며 이 경우 캐싱 과정을 동적으로 결정하는 특징을 갖는다.

## 2026-03-31 Tue
### 캐시 처리 단계
- 웹 캐시의 기본적인 동작은 단순하며, 예를 들어 임의의 `GET` 메시지에 대한 처리 단계는 다음과 같이 표현할 수 있다.
  1. 캐시는 네트워크로부터 도착한 요청 메시지를 읽어들인다.
  2. 메시지를 파싱하여 URL과 헤더 정보를 추출한다.
  3. 이에 대한 로컬 복사본이 있는지 검사하고, 없는 경우 사본을 받아와 로컬에 저장한다.
  4. 캐시된 사본이 충분히 신선한지 검증한 후, 신선하지 않은 경우 서버에 변경 사항을 질의한다.
  5. 캐시는 새로운 헤더와 캐시된 본문을 기반으로 응답 메시지를 생성한 후, 이를 클라이언트에게 반환한다.

## 2026-04-01 Wed
### 캐시와 신선도
- 캐시된 사본은 항상 서버의 문서와 일치함을 보장할 수 없으며, 일반적으로 문서들은 시간에 따라 변동된다.
  - 때문에 캐시는 오래된 데이터를 제공하지 않아야 하며, 캐시된 데이터가 서버의 데이터와 일치하도록 관리되어야 한다.
- HTTP는 어떤 캐시가 어떤 문서의 사본을 갖는지 서버가 알 필요 없이 신선도를 유지하도록 하기 위한 간단한 메커니즘인 문서 만료와 서버 재검사를 제공한다.
  - 예를 들어, **`Cache-Control`과 `Expires`와 같은 헤더를 통해 서버가 문서에 유효기간을 명시할 수 있도록 지원**한다.
  - `Expires`는 해당 문서가 만료되는 절대적인 시점을 명시하는 반면, `Cache-Control`은 `max-age` 옵션을 통해 만료되기까지의 시간을 명시한다.
  - 추가적으로, `Expires`는 `HTTP/1.0`의 개념인 반면 `Cache-Control`과 `max-age`는 `HTTP/1.1`의 개념에 해당한다.
- **클라이언트가 캐시된 리소스의 제공을 거부하지 않는 이상 캐시는 만료되지 않은 사본을 반환할 수 있으며, 만료시 서버에 변경점을 질의**해야 한다.

## 2026-04-02 Thu
### 서버 재검사란?
- **캐시된 문서의 만료는 해당 문서가 서버의 리소스와 항상 다른 것은 아니며, 다만 재검사할 시간이 되었음을 의미**한다.
  - 이렇듯 캐시가 대상 서버에게 문서의 변경 여부를 질의하는 일련의 과정을 서버 재검사라는 용어로 지칭한다.
- **서버 재검사시, 캐시는 원본이 변경되었다면 새로운 사본으로 업데이트하는 반면 변경되지 않은 경우 새로운 만료일로 갱신**한다.
  - 이러한 방식은 문서 만료시에만 검사를 유발하므로 신선하지 않은 컨텐츠를 사용자에게 제공하지 않으면서도 서버의 부담을 줄이는 좋은 방법에 해당한다.
- HTTP 프로토콜은 캐시가 신선하거나, 재검사로 인해 신선하다고 확신 가능한 사본을 반환하거나 에러 메시지를 반환하는 행동 중 하나를 수행할 것을 기대한다.

## 2026-04-03 Fri
### 조건부 메소드란?
- HTTP의 조건부 메소드는 재검사 과정의 효율을 높여주며, 예를 들어 HTTP는 서버에게 조건부 `GET` 요청을 전송할 수 있도록 허용한다.
  - 이러한 **조건부 메소드는 요청에 `If-` 접두사를 갖는 조건부 헤더를 추가하는 것으로 사용이 가능**하다.
  - 조건부 `GET`을 예로 들어, 서버는 클라이언트가 요청한 조건에 부합하는 경우에만 응답 본문에 리소스 정보를 포함시킨다.
  - 이렇듯 **조건부 메소드는 신선도 검사와 객체 반환을 하나의 요청으로 결합시키는 데에 의의**를 둔다.
- HTTP는 다섯 개의 조건부 요청 헤더를 정의하며, 그 중 가장 유용한 것 두가지는 다음과 같다.
  1. `If-Modified-Since`: 요청된 문서가 명시된 날짜 이후로 수정된 경우에만 요청을 처리한다.
  2. `If-None-Match`: `ETag`와 같이 문서에 대한 일련 변호가 다른 경우에만 요청을 처리한다.

## 2026-04-04 Sat
### If-Modified-Since 헤더를 활용한 날짜 기반 재검사
- **재검사 과정에서 가장 자주 사용되는 헤더이며, 해당 헤더가 포함된 요청에 대해 서버는 문서가 명시된 날짜 이후로 변경된 경우에만 본문을 포함**시킨다.
- 예를 들어, 해당 헤더에 명시된 날짜와 원본 문서의 수정 일자를 비교했을 때 서버는 다음과 같이 동작한다.
  1. 문서가 명시된 날짜 이후에 변경된 경우: 조건이 참이므로, `GET` 요청은 평범하게 처리되어 새로운 만료 날짜가 포함된 헤더와 새 문서가 반환된다.
  2. 문서가 명시된 날짜 이후에 변경되지 않은 경우: 조건이 거짓이므로, 서버는 `304 Not Modified` 응답을 반환하되 본문은 포함하지 않는다.
- 이 때, **서버가 `304`로 응답하는 경우 응답 헤더에는 새로운 만료 날짜 등 갱신이 필요한 헤더들이 포함**되어야 한다.
  - 예를 들어, `Content-Type`과 같은 헤더는 잘 변경되지 않으므로 명시할 필요가 없다.
- 해당 헤더는 서버가 반환하는 `Last-Modified` 헤더와 함께 동작하며 대상 서버는 제공되는 문서에 대해 마지막 변경 일시를 해당 헤더에 명시한다.
  - 캐시는 캐시된 문서를 재검사하기 위한 마지막 수정 일시로 서버가 반환한 `Last-Modified` 헤더의 값을 활용한다.
- 반면, 일부 웹 서버는 `If-Modified-Since` 헤더와 관련된 동작을 실제 날짜 비교가 아닌 문자열 비교로 처리하는 경우가 있다.
  - 이 경우 정확히 이 날짜 이후로 변경되었는지 질의하는 것이 아닌, 수정 일시를 일종의 일련 번호로 취급하여 일련 번호의 변경 여부만을 검증한다.

## 2026-04-12 Sun
### If-None-Match 헤더를 활용한 엔티티 태그 기반 재검사
- 다음과 같은 유형의 문서 변경은 날짜 기반의 재검사를 활용하기엔 무리가 있을 수 있다.
  1. 일정 시간마다 문서가 다시 쓰여지나, 실제 내용에는 아무런 변화가 없는 경우.
  2. 주석의 변경 등으로 인해 전세계의 캐시 변경을 유발하기에는 사소한 변경 사항이 가해진 경우
  3. 서버가 문서의 변경 일시를 정확하게 파악할 수 없는 경우
  4. 1초보다 적은 간격으로 갱신되는 문서를 제공하는 서버의 경우
- 반면, 문서가 변경되었을 때 해당 문서의 관리자는 문서의 `ETag`를 새로운 버전으로 표현할 수 있는 경우가 있다.
  - 이렇듯 **`ETag`가 변경되는 경우에 캐시는 새로운 문서의 사본을 검증하기 위해 해당 헤더를 활용하는 조건부 요청을 전송**할 수 있다.
- `ETag` 기반의 검증 방식은 `If-Modified-Since`의 처리 방식과 유사하며, 단지 변경을 감지하기 위한 대상으로 날짜가 아닌 `ETag`를 사용한다.
  - 반면, 캐시가 문서에 대한 여러 사본을 갖는 경우 `If-None-Match` 헤더에 쉼표로 구분된 여러 개의 `ETag`를 명시할 수 있다.

## 2026-04-13 Mon
### 강한 검사기와 약한 검사기
- 상술한 바에 따라, 캐시는 캐시된 버전이 서버가 갖고 있는 것과 비교했을 때 최신인지 검증하기 위해 날짜나 `ETag`를 활용할 수 있다.
  - 이 경우, **날짜와 `ETag`는 모두 캐시 검사기라는 용어로 지칭**된다.
- 그러나 서버는 캐시된 사본을 무효화시킬만큼 문서가 크게 변경되지 않은 경우, '이 정도면 수정 전과 동일한 문서다'라고 주장하고 싶을 수 있다.
- `HTTP/1.1`은 이를 위해 컨텐츠가 약간 변경되었음을 서버가 주장할 수 있도록 `약한 검사기` 기능을 지원하며, `W/` 접두사로 구분된다.
  - 예를 들어, `ETag: W/"v2.6"`과 같이 명시하는 것으로 약한 검사기를 구분할 수 있다.
- **강한 검사기는 대응되는 문서가 변경될 때마다 반드시 변경되는 반면, 약한 검사기는 문서에 유의미한 변경이 있을 때에만 변경**된다.

## 2026-04-14 Tue
### 엔티티 태그 또는 날짜 기반 재검사의 적용 기준
- **`HTTP/1.1` 클라이언트는 서버가 `ETag` 헤더를 반환했다면 반드시 `ETag` 검사기를 사용**해야 한다.
  - 반면, 서버의 응답에 `Last-Modified` 헤더만이 포함되었다면 날짜 기반 검사기를 사용할 수 있다.
- 그러나 **클라이언트는 서버가 두 헤더를 모두 사용 가능하도록 응답을 반환한 경우에는 두 재검사 정책을 모두 사용할 필요**가 있다.
- `HTTP/1.1` 서버는 가능하다면 `ETag` 검사기를 반환해야 하며, 이점이 있다면 약한 `ETag`를 반환할 것을 고려해볼 수도 있다.
  - 또한, 가능하다면 `Last-Modified` 값을 함께 반환하는 것을 고려해야 한다.
- 반면, **`HTTP/1.1` 기반 서버 또는 캐시가 `If-Modified-Since`와 `If-None-Match`를 동시에 요청받은 경우 두 조건을 모두 검증**해야 한다.
  - 두 조건에 모두 부합하는 경우에만 `304`를 반환해야 하며, 그렇지 못한 경우라면 `304`를 반환하지 않아야 한다.

### no-store와 no-cache 방식
- 두 방식은 모두 캐시가 검증되지 않은 캐시된 객체로 응답하는 것을 방지하며, 크게 다음과 같은 차이를 갖는다.
  1. `no-store`: 캐시가 응답의 사본을 생성하는 것을 금지하며, 캐시는 클라이언트에게 응답을 전달한 후 객체를 삭제한다.
  2. `no-cache`: 로컬 캐시 저장소에 객체를 저장할 수는 있으나, 서버와 재검사를 거치지 않고서는 클라이언트에 객체를 제공하지 않는다.
- 반면, `HTTP/1.0+`와의 하위호환성을 위해 `Pragma: no-cache` 헤더를 사용할 수 있다.
  - `HTTP/1.1` 기반 애플리케이션은 `HTTP/1.0`을 사용하는 애플리케이션에 대응해야하는 경우 외에는 `Cache-Control: no-cache`를 사용해야 한다.

  - 그러나 오래된 방식의 프록시는 암호화된 SSL 트래픽을 처리할 수 없었으며, 이에 터널을 사용하여 SSL 트래픽을 HTTP 커넥션으로 통과시킬 수 있도록 했다.
- SSL 터널은 암호화된 데이터를 HTTP 메시지에 담아 전송하며, 보안 처리가 적용된 웹 서버는 암호화된 SSL 트래픽을 그대로 전송 받는다.
- 이러한 터널은 HTTP가 아닌 트래픽이 방화벽을 통과할 수 있도록 우회하는 데에 유용하지만, 악의적인 트래픽이 유입되는 경로가 될 수도 있기에 주의해야 한다.
  - 이렇듯 터널 게이트웨이는 통신 중인 프로토콜이 적합한지 확인할 방법이 없으므로, 오용을 막기 위해 443과 같은 특정 포트만을 터널링하도록 허용해야 한다.

### 릴레이란?
- HTTP 릴레이는 HTTP 명세를 완전히 준수하지는 않는 간단한 프록시로, 커넥션을 맺은 후에는 그저 바이트를 전달하기만 하는 역할을 수행한다.
  - HTTP는 복잡한 편에 속하므로, 모든 헤더와 메소드 검증을 수행하지 않고 그저 바이트만을 전달하는 구현 방식이 유용할 때가 있다.
  - 이러한 데이터 전달은 구현이 쉽기에, 단순한 필터링이나 컨텐츠 변환 용도로 사용할 수도 있다.
- 반면, 단순한 릴레이를 구현했을 때 `Connection` 헤더를 정상적으로 처리하지 못할 경우 `keep-alive` 커넥션이 행에 걸릴 수 있음에 주의해야 한다.
  - 더 자세히 말하자면 `Connection` 헤더는 홉 간 헤더이나, 단순 릴레이가 이를 제거하지 않고 그대로 서버로 전달했을 때 문제가 발생할 수 있다.
- 때문에 단순한 형태의 릴레이를 구현하고자 하는 경우라면 이를 왜 사용하는지 신중히 고민할 필요가 있다.
  - 만약 필요에 의해 릴레이의 도입을 결정했다면, 여러 문제를 미연에 방지할 수 있도록 HTTP 명세를 정확히 준수하는 프록시를 사용하는 것이 적절하다.

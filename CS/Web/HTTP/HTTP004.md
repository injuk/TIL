### 웹 서버란?
- 웹 서버는 월드 와이드 웹의 일부로서, HTTP 요청을 처리하고 응답을 반환하는 역할을 수행한다.
  - 이 때, 웹 서버라는 용어는 물리적인 서버 자체와 웹 서버 소프트웨어 양쪽을 모두 지칭한다.
  - 반면, 웹 서버는 기능과 목적에 따라 다양한 규모로 구현될 수 있으나 모두 HTTP 요청을 받아 이를 처리한 후 클라이언트에게 응답한다는 공통점을 갖는다.
- 웹 서버는 HTTP나 그와 관련이 있는 TCP 처리를 구현하며, 주로 자신이 제공하는 리소스를 관리하기 위한 것과 웹 서버를 관리하기 위한 기능을 제공한다.
  - 이렇듯 **웹 서버는 HTTP 프로토콜을 구현하므로 TCP 커넥션 관리에 대한 책임을 운영체제와 나누어 갖는다**.
  - 예를 들어, 운영체제는 컴퓨터 시스템의 하드웨어를 관리하고 네트워크를 지원하며 파일 시스템과 프로세스 관리 기능을 제공한다.

### 클라이언트의 커넥션 수락하기
- 이미 맺어진 지속적인 커넥션이 있다면 클라이언트는 이를 재사용할 수 있으나, 그렇지 않은 경우 새로운 커넥션을 맺을 필요가 있다.
- **클라이언트가 웹 서버에 새로운 TCP 커넥션을 요청한 경우, 웹 서버는 커넥션을 맺고 이로부터 IP 주소를 추출하여 상대편 클라이언트를 확인**한다.
- 웹 서버는 커넥션을 원할 때에 거절하거나 닫을 수 있으므로, 구현에 따라서는 악의적인 것으로 알려진 커넥션을 즉시 닫도록 동작할 수도 있다.
  - 대부분의 웹 서버는 역방향 DNS를 사용하여 클라이언트의 IP 주소를 호스트 명으로 변환하도록 설정되어 있다.
  - 웹 서버는 이러한 클라이언트 호스트 명을 접근 제어와 로깅에 사용할 수 있으나, 이러한 lookup 작업은 웹 트랜잭션을 느려지게 하는 주요한 원인이기도 하다.
  - 때문에 대부분의 대용량 웹 서버는 호스트명 분석 기능을 명시적으로 비활성화하거나, 특정 컨텐츠에 대해서만 활성화하도록 설정한다.
- 웹 서버는 클라이언트의 사용자 이름을 얻기 위해 `ident` 프로토콜을 지원하는 경우가 있으며, 클라이언트가 이를 지원하는 경우 113 포트를 기반으로 동작한다.
  - 그러나 **`ident` 프로토콜은 HTTP 트랜잭션을 유의미하게 지연시키고, 안전하지 않기에 조직 내부가 아니라면 잘 사용되지 않는다**.

## 2026-02-22 Sun
### 요청 메시지 수신 단계
- 웹 서버는 커넥션에 도착한 데이터를 다음과 같은 순서로 읽어들여 요청 메시지 형태로 파싱한다.
  1. 요청줄은 공백으로 분리되고 CRLF로 끝나며, 메소드와 URI 및 버전 번호를 찾아 파싱한다.
  2. 헤더의 끝인 CRLF로 끝나는 빈 줄을 찾을 때까지 CRLF로 끝나는 모든 메시지 헤더를 찾아 파싱하여 헤더 목록을 구성한다.
  3. 요청 본문이 있다면 읽어들이되, 길이는 `Content-Length` 헤더에 따른다.
- 반면, 커넥션은 언제든지 무효화될 수 있기에 웹 서버는 이에 대비할 필요가 있다.
  - 예를 들어, **웹 서버는 파싱하여 이해 가능한 수준에 이르기까지 읽어들인 메시지의 일부를 메모리에 임시 저장**한다.

## 2026-02-23 Mon
### 웹 서버의 커넥션 입출력 처리 아키텍쳐
- 좋은 웹 서버는 수천 개의 커넥션을 동시에 열 수 있도록 지원하며, 각 커넥션은 전세계의 클라이언트 중 일부와 하나 이상 연결된다.
  - 웹 서버는 새로운 요청에 대비하고 있으며, 모든 커넥션의 처리 속도는 같지도 않다.
- 웹 서버는 각 커넥션을 처리하는 방법에 따라 서로 다른 아키텍쳐로 구성될 수 있으며, 이를 다음과 같이 구분할 수 있다.
  1. 단일 스레드: 한 번에 하나의 요청을 처리하며, 처리 도중 요청된 새로운 커넥션은 무시된다.
  2. 멀티프로세스 및 멀티스레드: 여러 요청을 동시에 처리할 수 있으나, 각 워커는 메모리를 소비하므로 최대 개수에 제한을 둔다.
  3. 다중 IO: 모든 커넥션은 감시되며, 커넥션의 상태가 변경될 때에만 적절한 처리가 적용된다.
  4. 다중 멀티스레드: 멀티스레딩과 다중 IO 방식을 결합한 것으로, 커넥션 상태 변경에 따른 처리는 각 스레드가 담당한다.

  - 반면, `OPTIONS`와 같은 일부 메소드는 본문을 허용하지만 필수적으로 요구하지는 않는다.
  - 물론 `GET`과 같이 본문 자체를 금지하는 메소드 역시 존재한다.
- 이렇듯 웹 서버는 자신이 전달 받은 HTTP 메시지의 내용에 따라 서로 다른 처리 방식을 적용하게 된다.

## 2026-02-26 Thu
### Docroot란?
- 웹 서버의 파일 시스템에 위치한 임의의 폴더를 웹 컨텐츠를 위해 예약할 경우, 이를 문서 루트 또는 `docroot`라는 용어로 지칭할 수 있다.
- 아파치 웹 서버를 예로 들어, `httpd.conf`에 `DocumentRoot`를 명시하는 것으로 문서 루트를 지정할 수 있다.
  - 예를 들어 `/usr/local/httpd/files`가 문서 루트로 명시될 수 있다.
  - 이 경우, 사용자가 `/image/something.png`를 요청할 경우 문서 루트에서 `image` 폴더를 찾아 반환하게 된다.
- 일반적으로, 웹 서버는 `../`와 같은 상대 경로에 의해 문서 루트를 벗어난 파일 시스템이 노출되지 않도록 동작한다.

### 가상 호스팅된 Docroot
- 가상 호스팅이 적용된 웹 서버는 각 사이트마다 분리된 문서 루트를 지정할 수 있으며, 이로 인해 하나의 웹 서버에서 여러 개의 웹 사이트를 호스팅할 수 있다.
- 가상 호스팅 웹 서버는 URI나 `Host` 헤더의 정보와 호스트 명을 활용하여 적절한 문서 루트를 결정한다.
  - 이를 통해 하나의 웹 서버 상에서 두 개의 사이트가 완전히 분리된 컨텐트를 제공하도록 호스팅될 수 있다.
- 대부분의 웹 서버는 가상 호스팅 기능을 간단히 지원하며, 아파치 웹 서버의 경우 `VirtualHost` 키워드를 활용한다.

### SSI란?
- SSI는 `Server-Side Includes`의 약자로, 서버가 임의의 리소스를 클라이언트에게 반환하기 전에 전처리를 적용하는 방식을 지칭한다.
  - 많은 웹 서버는 이러한 SSI 방식을 지원하며, 임의의 리소스가 SSI를 포함하도록 설정된다면 서버는 이에 대해 전처리를 적용한다.
- 서버는 SSI가 포함되도록 설정된 컨텐츠에 대한 요청을 다음과 같이 처리한다.
  1. 컨텐츠에 미리 정해진 변수나 임의의 패턴이 포함되었는지 검사한다.
  2. 포함된 경우, 실행 가능한 스크립트를 출력한 값을 각 변수나 패턴으로 치환하여 동적 컨텐츠를 생성한다.

### 응답 생성을 위한 MIME 타입 결정 방식
- 서버가 리소스를 식별한 후라면, 서버는 요청 메소드에 맞는 동작을 수행한 후 응답 메시지를 생성하여 반환한다.
- 이 때, 응답이 본문을 포함하는 경우 응답 메시지에는 `Content-Type`과 `Content-Length` 등의 헤더가 포함된다.
- 이는 바꿔 말해 서버가 MIME 타입을 결정할 수 있어야 함을 의미하며, 웹 서버는 MIME 타입 결정을 위해 다음과 같은 방식 중 하나를 사용할 수 있다.
  1. `mime.types` 방식: 가장 흔한 방식으로, 파일의 확장자를 통해 MIME 타입을 결정한다.
  2. 매직 타이핑 방식: 확장자가 없는 파일의 경우 파일을 직접 검사하여 알려진 패턴을 명시하는 매직 파일로부터 적절한 패턴을 찾는 느린 방식을 의미한다.
  3. 유형 명시 방식: 임의의 디렉토리 내의 파일은 확장자와 무관하게 미리 정의된 MIME 타입으로 취급하도록 설정하는 것을 의미한다.

### 프록시 연쇄에 따른 메시지의 추적을 위한 Via 헤더
- HTTP 인프라가 발전해감에 따라 프록시는 다양한 이유로 인해 점점 흔해졌으며, 이에 메시지의 흐름을 추적하는 것은 주요한 관심사로 떠오르게 되었다.
- 예를 들어 `Via` 헤더는 프록시나 게이트웨이 등의 중간 노드 정보를 나열하기 위해 사용되며, 중간 노드는 지속적으로 헤더 값의 끝에 추가된다.
  - 해당 헤더 필드는 메시지의 전달을 추적하고, 중간 노드들이 사용하는 프로토콜 정보를 확인하기 위해 사용될 수 있다.
  - 반면, 요청 메시지와 응답 메시지는 모두 프록시를 지나므로 모두 `Via` 헤더를 가지며 그 특성상 요청과 응답에 포함된 값은 언제나 반대가 된다.
- 응답 헤더에 포함되는 `Server`는 대상 서버에 사용되는 소프트웨어 정보를 명시하며, 이로 인해 프록시는 해당 헤더를 수정하지 않아야 한다.
  - 해당 헤더는 어디까지나 대상 서버의 정보를 명시하기 위함으로, 프록시는 대신 `Via` 헤더에 내용을 추가할 수 있다.
- 반면, 보안상 경유지 정보를 노출하지 않고자 하는 경우 그 경계에 포함된 호스트의 정보를 하나로 합쳐 `Via` 헤더에 명시할 수 있다.
  - 바꿔 말해 중간 노드들이 동일한 조직의 통제하에 있는 것이 아니라면 항목들을 합치지 않아야 한다.

### 프록시와 인증
- 프록시는 접근 제어 장치로 동작할 수 있으며, HTTP는 클라이언트가 적절한 자격 증명을 제출하지 않는 한 프록시에 의해 요청이 차단될 수 있다.
- 예를 들어, 클라이언트가 제한된 컨텐츠를 프록시에 요청한 경우 다음과 같은 절차가 진행된다.
  1. 프록시는 접근을 위한 자격 증명을 요청하는 `407 Proxy Authorization Required`로 응답한다.
  2. 이 때, 프록시는 응답에 `Proxy-Authenticate` 헤더에 자격 증명에 대한 제출 방법을 명시한다.
  3. 클라이언트는 상술한 헤더에 맞는 자격 증명을 스스로 얻어낸 후, `Proxy-Authorization` 헤더에 이를 명시하여 재요청한다.
  4. 프록시는 해당 자격 증명이 유효한 경우에만 프록시 연쇄를 통과시키며, 그렇지 않은 경우 다시 `407`로 응답한다.
- 반면, **프록시 인증은 인증에 참여하는 프록시가 연쇄상 여럿 존재할 경우에는 잘 동작하지 않는다는 한계**를 갖는다.

### 프록시와 상호운용성
- `HTTP/1.1`은 메소드 확장이나 사용자 정의 헤더를 허용하므로, 프록시는 자신이 이해하지 못하는 메소드와 헤더를 그대로 유지하도록 동작해야 한다.
- 또한, 클라이언트는 `OPTIONS` 메소드를 전달하는 것으로 대상 서버가 어떤 메소드를 제공하는지에 대한 정보를 얻어낼 수 있다.
- 예를 들어 `HTTP/1.1`의 경우, 서버에 어떤 메소드가 지원되는지 서술하기 위해 `Allow` 헤더에 서버에서 지원하는 메소드의 목록을 명시할 수 있다.
  - 반면, 클라이언트에 의한 요청 메시지에 `Allow` 헤더가 포함된 경우 해당 리소스에 대해 클라이언트가 희망하는 메소드를 의미하게 된다.
  - 이에 대상 서버는 해당 리소스에서 실제로 지원하는 메소드만을 `Allow` 헤더에 명시하여 응답하며, 클라이언트의 요청 메소드를 모두 지원해야 할 의무는 없다.
  - **프록시의 경우, `Allow` 헤더에 명시된 모든 메소드를 이해할 수 없다고 하더라도 해당 헤더의 내용을 수정하지 않아야 한다**.

# HTTP
## 2026-06-16 Tue
### HTTP 엔티티의 필요성
- HTTP는 매일 수많은 미디어 객체를 전송하며, 또한 다음과 같이 메시지의 전송과 식별 및 추출과 처리 과정이 올바를 것을 보장한다.
  1. Content-Type과 Content-Language는 올바르게 식별되므로 HTTP 클라이언트는 컨텐츠를 올바르게 처리할 수 있다.
  2. Content-Type과 Content-Encoding 헤더를 통해 객체는 올바르게 압축이 풀릴 수 있다.
  3. 엔티티 검사기와 캐시 만료 제어를 기반으로 최신의 객체를 사용하게 된다.
  4. 내용 협상을 위한 Accept 헤더에 기반하여 사용자의 요구를 만족시킨다.
  5. 객체는 네트워크 간에서 빠르고 효율적으로 이동된다.
  6. 객체는 중도에 조작되는 일 없이 온전하게 전달된다.
- 상술한 모든 항목을 보장하기 위해 HTTP는 적절한 라벨링이 완료된 객체인 엔티티를 사용한다.

## 2026-06-17 Wed
### HTTP 메시지와 HTTP 엔티티의 관계
- 인터넷이라는 운송 시스템에서, HTTP 메시지를 컨테이너에 비유한다면 HTTP 엔티티는 실질적인 화물로 비유할 수 있다.
  - 이러한 HTTP 엔티티는 HTTP 본문과 혼동되지 않아야 하며, HTTP 본문과 `Contenxt-XXX` 헤더와의 조합으로 이해할 수 있다.
- HTTP 엔티티를 구성하는 엔티티 헤더는 크게 다음과 같이 분류할 수 있다.
  1. Content-Type: 엔티티에 의해 전달된 객체의 유형이다.
  2. Content-Length: 전달되는 메시지의 길이 또는 크기를 의미한다.
  3. Content-Language: 전달되는 객체와 가장 잘 대응되는 자연어를 의미한다.
  4. Content-Encoding: 전달되는 객체에 압축 등을 위해 가해진 변형을 의미한다.
  5. Content-Location: HTTP 요청이 있던 시점을 기준으로 객체의 또다른 위치를 의미한다.
  6. Content-Range: 해당 엔티티가 부분 엔티티라면, 해당 헤더를 통해 엔티티가 전체의 어느 부분에 포함되는지 정의할 수 있다.
  7. Content-MD5: 엔티티 본문에 대한 체크섬값을 의미한다.
  8. Last-Modified: 서버에서 해당 컨텐츠가 생성되었거나, 마지막으로 수정된 일시를 의미한다.
  9. Expires: 해당 엔티티의 데이터가 더 이상 신선하지 않다고 가정되기 시작하는 일시를 의미한다.
  10. Allow: 이 리소스에 대해 어떤 HTTP 요청 메소드가 허용되는지 명시한다.

## 2026-06-18 Thu
### 엔티티 본문이란?
- 엔티티 본문은 가공되지 않은 데이터만을 포함하며, 다른 정보들은 모두 헤더에 명시한다.
  - 이는 본문이 HTTP에 의해 가공되지 않음을 의미하며, 사용자의 의도에 따라 이미 가공된 데이터일 수도 있다.
  - 단, 데이터가 가공되었다면 그 사실을 상술한 엔티티 헤더에 명시하는 원리로 이해할 수 있다.
- 이렇듯 본문은 가공되지 않은 날 것의 데이터이며, 자신을 설명하지 않으므로 엔티티 헤더는 본문 데이터를 어떻게 해석할지에 대한 메타데이터로서 기능할 수 있다.
  - 예를 들어, Content-Encoding 헤더가 데이터의 압축 및 인코딩 여부에 대해 알려준다.
- 엔티티 본문은 컨텐츠의 종류와 무관하게, 반드시 헤더 필드의 끝을 의미하는 빈 CRLF 줄 다음에 위치한다.

## 2026-06-19 Fri
### Content-Length 헤더의 용도
- Content-Length 헤더는 엔티티 본문의 크기를 바이트 단위로 표현하며, 인코딩 여부와 관계 없이 전체 크기를 의미한다.
  - 예를 들어 텍스트가 gzip으로 압축되었다면, 해당 헤더는 압축된 후의 크기를 명시하게 된다.
- Content-Length 헤더는 메시지가 청크 인코딩을 사용하지 않는 이상 엔티티 본문을 포함하는 메시지에 필수 요소이다.
- Content-Length 헤더는 서버 충돌로 인해 메시지가 잘렸는지 감지하거나, 지속 커넥션을 공유하는 메시지를 올바르게 분할하는 데에 사용될 수 있다.
  - 초기 HTTP는 커넥션이 닫힐 때 메시지의 끝을 인식했으나, 이 방식은 커넥션 정상 종료와 서버 충돌로 인한 오류를 구분할 수 없었다.
- HTTP/1.1 명세의 경우, 어떠한 헤더도 인코딩되지 않은 본문의 길이를 보내기 위해 사용될 수는 없다.
  - 이는 클라이언트가 자신의 디코딩 과정에 문제가 없는지 검증하는 것을 어렵게 만드는 한계를 갖는다.

## 2026-06-20 Sat
### 엔티티 본문 길이 판별 규칙
- 본문을 가질 수 없는 유형의 HTTP 메시지라면 본문 계산을 위한 Content-Length 헤더가 무시될 수 있다.
  - 이 경우, 해당 헤더는 추가 정보로서 기능하며 실제 본문의 길이와는 무관하다.
  - 예를 들어 HEAD 요청에 대한 응답은 본문을 갖지 않음에도 Content-Length 헤더는 포함한다.
  - 이는 HEAD의 용도가 GET 요청에 대한 응답에서 본문만을 제거한 것이기 때문이므로, 1xx나 204 및 304 역시 정보성 Content-Length를 갖는다.
- 메시지가 Transfer-Encoding 헤더를 포함한다면 엔티티는 커넥션 종료가 아니라면 0 바이트 청크라는 특수 패턴으로 끝나야 한다.
- 메시지가 Content-Length 헤더를 갖고, Transfer-Encoding 헤더를 갖지 않는다면 해당 헤더는 본문의 길이를 의미한다.
  - 바꿔 말해, Transfer-Encoding 헤더를 포함하는 메시지에 대해 클라이언트는 Content-Length 헤더를 반드시 무시해야 한다.
- 메시지가 multipart/byteranges 미디어 타입을 사용하고, 엔티티 길이가 정의되지 않았다면 멀티파트 메시지의 각 부분이 스스로 크기를 정의한다.
  - 이러한 멀티파트 유형은 자신의 크기를 스스로 결정 가능한 유일한 본문 유형에 해당한다.
- 상술한 어떠한 규칙에도 해당하지 않는 경우, 엔티티는 커넥션 종료시에 끝난다.
- HTTP/1.0 애플리케이션과의 호환성 유지를 위해 엔티티 본문을 갖는 HTTP/1.1 요청은 반드시 유효한 Content-Length 헤더를 가져야한다.
  - HTTP/1.1 명세는 Content-Length 헤더 누락시 400 또는 411 응답을 반환할 것을 조언한다.

### 엔티티 요약과 MD5
- HTTP는 불완전한 프록시 등의 여러 이유에서 메시지의 일부분이 중간에 변경될 수 있다.
- 이를 방지하기 위해 엔티티를 최초로 생성하는 송신자는 데이터에 대한 체크섬을 생성할 수 있다.
  - 반면, 수신자는 모든 엔티티의 변경을 감지할 수 있도록 이러한 체크섬으로 기본적인 검사를 수행할 수 있다.
- 예를 들어, Content-MD5 헤더는 서버가 엔티티 본문에 MD5 알고리즘을 적용한 결과를 명시하기 위해 사용된다.
  - 이러한 방법은 본문과 헤더 전체가 교체되는 악의적인 공격에는 내성이 없으며, 단지 의도치 않은 변형만을 감지할 수 있음에 주의해야 한다.
- MD5는 문서의 위치를 빠르게 식별하고 중복 저장을 방지하기 위한 해시로도 사용될 수 있는 등 유용하지만 아쉽게도 Content-MD5 헤더는 자주 사용되지 않는다.

## 2026-06-22 Mon
### Content-Type 헤더와 MIME
- Content-Type 헤더는 엔티티 본문의 MIME 타입을 기술하는 데에 사용된다.
  - 이러한 MIME은 전달되는 데이터 매체의 기저 형식을 지칭하는 표준이다.
  - 클라이언트 애플리케이션은 이러한 MIME 타입을 통해 컨텐츠를 적절히 처리할 수 있다.
- MIME 타입은 주 미디어 타입과 부 타입, 그리고 이 둘을 구분하는 슬래쉬 기호로 작성된다.
- 이 때, **Content-Type 헤더가 엔티티 본문의 미디어 타입을 명시한다는 점에 주의**한다.
  - 즉, 엔티티가 인코딩되었더라도 해당 헤더는 인코딩 전의 본문 유형을 명시해야 한다.

### 텍스트 매체를 위한 문자 인코딩 방식
- Content-Type 헤더는 내용을 더 자세히 명시하기 위한 선택적인 변수를 지원한다.
  - 예를 들어, 엔티티 비트 집합을 글자로 변환하기 위해서는 `charset` 매개변수를 사용할 수 있다.
- `charset` 매개변수는 `Content-Type: text/html; charset=iso-8859-4` 형태로 사용된다.
  - 이는 기본 MIME이 text/html이되, 글자에 대한 캐릭터 셋이 `iso-8859-4`임을 의미한다.

### 멀티파트 미디어 타입
- 멀티파트 메시지는 서로 붙어있는 여러 메시지를 포함하되, 하나의 복합 메시지 형태로 전송된다.
  - 각 구성 요소는 자신에 대해 스스로를 서술하는 헤더를 포함해야 한다.
- 이러한 멀티파트 메시지는 여러 구성요소들이 이어지고, 문자열 하나로 서로의 경계를 식별하는 형태로 작성된다.
- HTTP의 경우 멀티파트 본문도 지원하지만, 일반적으로는 폼을 제출할 때와 컨텐츠의 일부를 잘라 반환하는 범위 응답에 사용된다.
  - 멀티파트 폼을 예로 들어, 가변 길이 텍스트 필드와 업로드된 객체는 각각 멀티파트 본문을 구성하는 파트로 취급되어 전송된다.
  - 때문에 멀티파트 본문은 서로 다른 종류와 값으로 채워지는 폼을 허용할 수 있다.
  - 반면, 범위 전송을 위한 멀티파트의 경우 `Content-Range` 등의 별도 헤더를 포함한다.

### 컨텐츠 인코딩이란?
- HTTP 애플리케이션은 필요에 따라서는 컨텐츠 전송 전에 인코딩을 우선 진행할 수도 있다.
  - 예를 들어, 거대한 HTML 문서를 클라이언트에게 전송하기 전에 우선 압축을 진행할 수도 있다.
  - 이러한 압축 과정 역시 일종의 인코딩에 해당하며, 또는 인가되지 않은 사용자의 접근을 막기 위해 암호화를 적용할 수도 있다.
- 어떠한 종류의 인코딩이든 송신자가 컨텐츠에 적용하게 되며, 인코딩 완료된 데이터는 본문에 담겨 수신자에게 전송된다.

## 2026-06-26 Fri
### 컨텐츠 인코딩 과정
- 일반적인 컨텐츠 인코딩 과정은 다음과 같이 요약할 수 있다.
  1. 웹 서버는 원본 Content-Type과 Content-Length 헤더를 수반하는 원본 응답 메시지를 생성한다.
  2. 컨텐츠 인코딩 서버는 메시지를 인코딩하며, 이 경우 Content-Type은 유지되지만 Content-Length는 변경된다.
  3. 컨텐츠 인코딩 서버는 Content-Encoding 헤더를 메시지에 명시한 후 수신자에게 전송한다.
  4. 수신자는 인코딩된 메시지를 수신하고, 이를 적절히 디코딩한 후 원본을 확인할 수 있게 된다.
- 이 때, 상술한 바와 같이 Content-Type 헤더가 메시지에 존재할 수 있고 마땅히 그래야한다는 점에 주의해야 한다.
  - 이 헤더는 엔티티 본래의 포맷을 명시하며, 디코딩된 엔티티를 검증하기 위해 필수적인 정보이다.
  - 반면, Content-Length 헤더는 인코딩된 본문의 길이를 나타낸다는 점에 주의해야 한다.

## 2026-06-27 Sat
### 컨텐츠 인코딩 유형
- HTTP는 기본적으로 몇 가지의 표준 컨텐츠 인코딩을 정의하며, 추가적으로 확장 인코딩을 활용한 인코딩 추가 역시 허용한다.
  - 이 때, 인코등인 각 컨텐츠 인코딩 알고리즘에 고유한 토큰을 할당하는 IANA와 같은 기관에 의해 표준화된다.
  - Content-Encoding 헤더는 이러한 표준화된 값을 명시하는 것으로 인코딩에 적용된 알고리즘을 명시한다.
- 이 때, 특히 자주 사용되는 컨텐츠 인코딩 토큰은 다음과 같이 정리해볼 수 있다.
  1. `gzip`: 엔티티에 GNU zip 인코딩이 적용된 경우
  2. `compress`: 엔티티에 동명의 유닉스 파일 압축 프로그램이 적용된 경우
  3. `deflate`: 엔티티가 zlib 포맷으로 압축된 경우
  4. `identity`: 엔티티에 어떠한 인코딩도 수행되지 않았으며, **Content-Encoding 헤더가 누락된 경우의 기본 값에 해당**한다.

### Accept-Encoding 헤더란?
- 반면, 서버가 인코딩한 방법으로 모든 클라이언트가 디코딩할 수 있다고 기대할 수는 없다.
  - 예를 들어, 서버가 인코딩한 도구를 클라이언트가 갖지 않는다면 클라이언트는 이를 디코딩할 수 없다.
- 서버가 클라이언트가 지원하지 않는 인코딩을 적용하는 것을 방지하기 위해 클라이언트는 지원 인코딩 목록을 Accept-Encoding 헤더를 통해 전달한다.
  - 반면, HTTP 요청에 해당 헤더가 포함되지 않았다면 서버는 클라이언트가 모든 인코딩 방식을 허용하는 `Accept-Encoding: *` 상태인 것으로 간주한다.
- Accept-Encoding 헤더는 쉼표로 구분된 컨텐츠 인코딩 유형을 명시하며, 자주 사용되는 예시는 다음과 같다.
  1. `Accept-Encoding: gzip, deflate`
  2. `Accept-Encoding: gzip, deflate;q=0.6, *;q=0.8`
  3. `Accept-Encoding: *`
- 이 때, 클라이언트는 인코딩의 품질 값인 `q` 매개변수를 전달하여 선호도를 나타낼 수 있다.
  - 예를 들어, 가장 원치 않는 방식은 `0.0`인 반면 가장 원하는 방식은 `1.0`으로 표기한다.

### 전송 인코딩이란?
- 컨텐츠 인코딩은 컨텐츠의 유형과 긴밀하게 연관되어 있으며, 예를 들어 gzip은 텍스트 파일 압축에 사용되지만 이미지 압축에는 적절하지 않다.
- 이에 전송 인코딩 개념이 제안되었으며, 이 역시 엔티티 본문에 적용되는 가역적인 변환이지만 전체 메시지에 대해 인코딩을 적용한다는 점에서 차이가 있다.
  - 반면, 컨텐츠 인코딩은 메시지의 엔티티 부분만을 인코딩하므로 전송 인코딩은 메시지 전체의 구조를 바꾼다는 점에서도 큰 차이를 갖는다.
- 이 때, 전송 인코딩은 안전한 전송을 가능케할 수 있도록 Transfer-Encoding 헤더와 TE 헤더를 도입하여 사용한다.

### 전송 인코딩과 관련된 두가지 문제
- HTTP 상에서 전송된 메시지 본문이 문제를 일으킬 수 있는 상황은 크게 다음과 같은 두가지 예시를 들 수 있다.
  1. 메시지 본문의 최종 크기를 판단하지 못했음에도 서버가 데이터 전송을 시작하는 경우
  2. 공용 네트워크로 메시지를 전송하기 전에 전송 인코딩을 통해 알아볼 수 없도록 암호화하는 경우
- 첫 번째 문제의 경우, 이렇게 동작하는 서버는 미리 정해진 문자열을 종결 신호로 사용할 수도 있으나 이러한 방식은 지속 커넥션을 망가트린다. 
- 두 번째 문제의 경우, 이러한 자체적인 보안 방식보다는 SSL과 같은 유명한 전송 계층 보안 방식을 사용하는 것이 바람직하다.

## 2026-07-01 Wed
### 전송 인코딩을 위한 두 가지 헤더
- 전송 인코딩을 위해 정의된 헤더는 상술했듯 다음과 같은 두 종류에 해당한다.
  1. Transfer-Encoding: 안전한 전송을 위해 어떤 종류의 인코딩이 메시지에 적용되었는지 명시하기 위해 서버가 사용한다.
  2. TE: 어떤 확장 전송 인코딩을 사용할 수 있는지 서버에게 알려주기 위해 클라이언트가 사용한다.
- 예를 들어, 클라이언트는 자신이 청크 인코딩 메시지와 메시지 끝의 트레일러를 받아들일 수 있음을 TE 헤더에 명시하여 알릴 수 있다.
  - 이 경우, `TE: trailers, chunked`와 같은 형태의 헤더를 사용하게 된다.
- 반면, 서버는 메시지가 청크 인코딩을 기반으로 전송 인코딩되었음을 `Transfer-Encoding: chunked`와 같이 명시할 수 있다.
  - 이 경우, 기본 헤더 뒤의 메시지 구조가 변경될 수 있음에 주의해야 한다.

## 2026-07-02 Thu
### 청크 인코딩이란?
- 청크 인코딩은 메시지를 여러 개의 청크로 쪼개는 방식으로, 이 경우 서버는 각 청크를 순차적으로 전송하는 식으로 동작한다.
- 이 때, 청크 인코딩은 본문이 동적으로 생성되므로 메시지 전송 전에 전체 본문의 크기를 알 필요가 없다는 특징을 갖는다.
  - 서버는 그 중 일부인 청크를 버퍼에 담아 청크 별 크기를 명시하여 전송하며, 이 과정을 전체 메시지가 전송될 때까지 반복한다.
- 반면, 청크 인코딩은 전송 인코딩의 한 형태이므로 본문이 아닌 메시지 자체의 속성에 해당한다.
  - 이는 멀티파트 인코딩이 본문의 속성인 특징과 비교했을 때 차이가 있다.

## 2026-07-03 Fri
### 청크와 지속 커넥션
- 클라이언트와 서버 간의 커넥션이 지속적이지 않다면, 클라이언트는 본문의 크기를 알 필요가 없다.
  - 이는 서버가 커넥션을 닫을 때까지 전송된 데이터를 본문으로 간주할 수 있기 때문이다.
- 반면, 지속 커넥션의 경우 본문을 쓰기 전에 Content-Length 헤더를 명시하지 않는다면 클라이언트가 본문의 끝을 인식할 수 없다.
  - 그러나 컨텐츠가 서버로부터 동적으로 생성되는 경우에는 전송 전에 전체 크기를 알 수 없다는 문제가 발생한다.
- 청크 인코딩은 서버가 본문을 여러 개의 청크로 쪼개어 보낼 수 있도록 한다는 점에서 이러한 문제를 해결하는 데에 사용될 수 있다.
  - 이 경우, 서버는 크기가 0인 청크로 본문 종료를 클라이언트에게 알린 후 커넥션을 열린 상태로 유지하여 다음 통신에 대비한다.
- 청크 인코딩된 메시지는 다른 HTTP 메시지와 마찬가지로 응답 헤더 블록으로 시작하며, 청크의 스트림이 이어지는 구조를 갖는다.
  - 각 청크는 길이 정보와 청크 자체에 대한 데이터를 담으며, 이 때 길이는 16진수 형태로 명시된다.
  - 반면, 청크 데이터는 바이트 단위로 측정되며, CRLF와 같은 문자열은 길이에 포함하지 않는다.
- 클라이언트 역시 청크 인코딩된 데이터를 전송할 수 있으나, 서버가 청크 인코딩을 받아들일 수 있을지 알 수 없기에 서버로부터의 411 응답에 대비해야 한다.
